# /testing-qa-detailed-design

> The Gate Definition step of a testing-qa activity: define which review gates the development process requires, as the project's single senior quality-assurance expert — not a piece of testing or quality-control work.

**Arguments** — `$ARGUMENTS` is `<component_id> <activity_id>` (two space-separated tokens; `component_id` is typically empty — this is a project-level activity, not a per-component one). Parse once; do not swap them. Work lands on the shared activity branch `activity/<activity_id>` and its single PR — you are contributing commits to a PR that already exists (or open it if this is the first phase). Do NOT open a second PR.

**Agent + skills.** Work to the standard of the **`qa-engineer`** agent (`.claude/agents/qa-engineer.md`). Follow **[[the-method-testing]]** and **[[the-method-project-state]]** for all reading/updating of `.aiarch/state/project.json`.

**Goal / intention.** Per ch. 9, "most teams incorrectly refer to their quality control and testing activities as quality assurance (QA)," but "true QA has little to do with testing. It typically involves a single, senior expert who answers the question: What will it take to assure quality?" — an answer that "must include how to orient the entire development process to assure quality, how to prevent problems from ever happening, and how to track the root causes of problems and fix them," and ch. 9 notes "the QA person is sometimes responsible for designing the process and authoring procedures for key phases." Ch. 14 sharpens this into a concrete quality-assurance activity: "Actively engage a true quality-assurance person. Have that person review the development process, tune it to assure quality, and create a process that is both effective and easy to follow" — one that "support[s] investigation and elimination of the root cause of defects or, even better, ... proactively prevent[s] problems from happening in the first place." Ch. 14 also calls for structured, role-based reviews at both the service and system level. This step is where that senior QA expert does that process design up front: deciding which review gates this project's development process requires — where human escalation is mandatory, which artifacts need structured review with designated roles, and what binary criteria each gate checks — before construction is underway. "Done" means the process now has defined, meaningful gates a developer or the Manager can check against. This step must NOT write or review any component's contract, design, or code (that is not QA's job); must NOT run tests or build a harness (test-engineer/software-tester's job — see [[the-method-testing]]); and — per Löwy, "the presence of a QA person is a sign of organizational maturity," this is a process artifact, not a testing one — must NOT itself decide the project's binding review policy: it proposes which gates the process needs, but the Manager alone records the actual enforced `.reviewPolicy`. Recording that policy is a separate, Manager-owned verb; the QA agent never writes it.

## Steps

> **State changes go through the `aiarch-state` MCP tools, not hand-edits.** Where a step below says to record a service contract, phase artifact, or testing artifact and "commit onto branch", do it with the matching tool — `recordServiceContract` / `recordPhaseArtifact` / `recordTestingState` — and finish with `publishDraft`. Do **not** hand-edit `.aiarch/state/project.json` or run `git` for state; only source/doc **files** (code, docs) are git-committed by you. See [[the-method-project-state]].

1. **Read what you need** from `.aiarch/state/project.json` per [[the-method-project-state]]: the activity, the current activity list and system design (to see what kinds of activities and components the process must gate), and any existing quality-process artifacts from earlier in the project.
2. **Produce** the phase artifact: the gate-definition note — naming which review gates the development process requires (which activity types/phases need human escalation, which artifacts need structured review with designated roles, and the binary criteria each gate checks) — recorded into the phase-artifacts store per [[the-method-project-state]] and committed onto branch `activity/<activity_id>`. Do NOT write `.reviewPolicy` yourself; that is a Manager-owned verb applied separately from this note.
3. **Verify** (only your own output; fast checks): every proposed gate has a binary, checkable criterion and a named artifact/phase it applies to; the note contains no component contract, design, code, or test content; the note does not itself claim to be the enforced review policy.
4. **Stop.** Do not mark phase status (the Manager owns that) and do not merge. Leave the PR open for the gate.
