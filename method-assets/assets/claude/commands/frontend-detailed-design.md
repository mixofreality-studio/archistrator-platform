# /frontend-detailed-design

> The Design step of a frontend activity: produce this UI surface's design concept — layouts, component choices, flows, states — and hand it to review before construction begins.

**Arguments** — `$ARGUMENTS` is `<component_id> <activity_id>` (two space-separated tokens; `component_id` may be empty for non-component activities). Parse once; do not swap them. Work lands on the shared activity branch `activity/<activity_id>` and its single PR — you are contributing commits to a PR that already exists (or open it if this is the first phase). Do NOT open a second PR.

**Agent + skills.** Work to the standard of the **`ui-designer`** agent (`.claude/agents/ui-designer.md`). There is no dedicated UI-design Method skill — this step's design work is scoped by the agent's own remit. Once produced, the design concept is reviewed via **[[the-method-review-routing]]** (founder/architect-user approval plus `ux-reviewer`, `product-manager`, and `system-architect`), not amended or frozen unilaterally. Follow **[[the-method-project-state]]** for all reading/updating of `.aiarch/state/project.json`.

**Goal / intention.** Per ch. 14's scope-versus-effort model, Löwy places "the services detailed design or Clients user interface" in the same tier of the design hierarchy: narrower in scope than architecture but far more time-consuming, running to weeks of work even for a handful of interacting parts. The book has no Appendix-B-style rulebook for UI-design method the way it does for service contracts, so the nearest real anchor is the chapter's own hand-off doctrine: a designer with authority over one piece of work produces a concrete design before anyone builds against it, and that design is never handed straight to construction — it goes to review, and only a reviewed, approved design is safe to construct against. This step is that discipline applied to one UI surface: produce the design concept — the layouts, component choices, flows, and states — that covers every core use case and persona journey captured in Requirements, and stop short of writing code. "Done" means a design concept exists that traces to Requirements and is ready for [[the-method-review-routing]] to gate; it does not mean the concept has been approved, and it certainly does not mean any surface has been built. This step must NOT write production UI code, must NOT change the committed system design, and must NOT let the concept proceed to construction unreviewed.

## Steps

> **State changes go through the `aiarch-state` MCP tools, not hand-edits.** Where a step below says to record a service contract, phase artifact, or testing artifact and "commit onto branch", do it with the matching tool — `recordServiceContract` / `recordPhaseArtifact` / `recordTestingState` — and finish with `publishDraft`. Do **not** hand-edit `.aiarch/state/project.json` or run `git` for state; only source/doc **files** (code, docs) are git-committed by you. See [[the-method-project-state]].

1. **Read what you need** from `.aiarch/state/project.json` per [[the-method-project-state]]: the activity, this surface's UX requirements from the Requirements phase, its entry (Client/SPA container, relationships to the Managers/Engines it calls) in the committed system design, and the core use cases and personas it must support.
2. **Produce** the phase artifact: the UI design concept for this surface — per-use-case/persona screen flows, layout, component selection, and states — recorded to `.aiarch/state/project.json` → `.phaseArtifacts.uiDesign[surface]` via `recordPhaseArtifact` per [[the-method-project-state]], then committed onto branch `activity/<activity_id>` and handed to review via [[the-method-review-routing]].
3. **Verify** (only your own output; fast checks): every screen, flow, and state in the concept traces to a use case or persona captured in the surface's UX requirements — no invented scope; no production UI code exists yet; the committed system design and any frozen service contracts are unchanged.
4. **Stop.** Do not mark phase status (the Manager owns that) and do not merge. Leave the PR open for the gate.
