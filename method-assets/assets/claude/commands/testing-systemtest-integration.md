# /testing-systemtest-integration

> The Regression & Sign-off step of a testing-systemtest activity: run the regression harness across the just-exercised system, confirm nothing destabilized, and record the sign-off note this terminal activity's exit depends on.

**Arguments** — `$ARGUMENTS` is `<component_id> <activity_id>` (two space-separated tokens; `component_id` is typically empty — System Testing is a system-level activity, not a per-component one). Parse once; do not swap them. Work lands on the shared activity branch `activity/<activity_id>` and its single PR — you are contributing commits to a PR that already exists (or open it if this is the first phase). Do NOT open a second PR.

**Agent + skills.** Work to the standard of the **`software-tester`** agent (`.claude/agents/software-tester.md`). Follow **[[the-method-testing]]** and **[[the-method-project-state]]** for all reading/updating of `.aiarch/state/project.json`.

**Goal / intention.** Grounded honestly: neither ch. 11 nor ch. 13 narrates regression execution or sign-off procedure in prose — both are project-design worked examples, focused on scheduling and staffing, not on what happens inside a testing activity. What they do establish, and what this step is built on, is structural: ch. 13's TradeMe activity table lists the Regression Test Harness (owned by a developer, not the test engineer) as its own tracked activity feeding into System Testing, and both worked examples place System Testing as the last coding-adjacent activity before the project can close out — ch. 13's table runs directly from "System Testing" (10 days) into "System Rollout." Nothing in the project is scheduled to catch a late destabilization after this point, which is why this activity's own doctrine ([[the-method-testing]]) has the tester — not the harness's developer-owner — run that regression harness continuously through System Testing and report destabilization the moment it happens, rather than trusting it silently. This step is where that responsibility closes out: execute the regression harness one more time against the system as Use-Case Execution left it, and produce the note that either affirms the system is clean or names the specific open defects (by severity) still blocking sign-off. "Done" means a regression run is recorded and a sign-off note is written; it must NOT fix any defect (that is a developer's job, in a follow-on hardening activity), must NOT quietly close or downgrade an open defect to manufacture a clean sign-off, and must NOT invent new use cases or scenarios — only run what already exists and report on it honestly.

## Steps

1. **Read what you need** from `.aiarch/state/project.json` per [[the-method-project-state]]: the activity, the System Test Harness, and the full test-run and defect history recorded so far in this activity (from the Smoke Pass and Use-Case Execution phases).
2. **Produce** the phase artifact: execute the developer-owned Regression Test Harness against the integrated system and record the run into the testing-state's test-run history, filing any newly observed regression as a defect; then write the regression-and-sign-off note — either affirming the system is clean, or naming the still-open critical/high defects blocking sign-off — into the phase-artifacts store per [[the-method-project-state]]. Commit onto branch `activity/<activity_id>`.
3. **Verify** (only your own output; fast checks): the regression run references the real harness and the real defect list, not an invented summary; the sign-off note's severity accounting matches the actual open defects — no open critical/high defect is glossed over; no defect was closed or downgraded by this step and no product code was changed.
4. **Stop.** Do not mark phase status (the Manager owns that) and do not merge. Leave the PR open for the gate.
