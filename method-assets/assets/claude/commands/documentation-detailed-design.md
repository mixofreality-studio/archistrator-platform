# /documentation-detailed-design

> The Outline step of a documentation activity: decide what documents this project needs, which committed design artifacts and diagrams each one must reuse or reference, and where each will live, before any prose or diagram is authored.

**Arguments** — `$ARGUMENTS` is `<component_id> <activity_id>` (two space-separated tokens; `component_id` may be empty for non-component activities). Parse once; do not swap them. Work lands on the shared activity branch `activity/<activity_id>` and its single PR — you are contributing commits to a PR that already exists (or open it if this is the first phase). Do NOT open a second PR.

**Agent + skills.** Work to the standard of the **`system-architect`** agent (`.claude/agents/system-architect.md`). Follow **[[the-method-architecture]]** and **[[the-method-project-state]]** for all reading/updating of `.aiarch/state/project.json`.

**Goal / intention.** Löwy does not devote a chapter to technical writing, and Appendix C has no "documentation" checklist section — where he is silent, this step grounds in the nearest real principle he does state repeatedly: the architect owns and communicates the design. Per ch. 3, "having clear, consistent nomenclature for components in your architecture and their relationship... improves communication because you can now convey your design intent to other architects or developers," and diagrams exist because "diagrams allow you to take advantage of [visual processing] to communicate ideas to your audience" — ch. 3's own rule of thumb is that a nested "if" in a use case is the signal you must draw it as an activity diagram, not describe it in prose. Per ch. 4, the call chain and sequence diagrams that validate a design are not one-off scratch work: "if you are going to produce them for the detailed design, you might as well produce them first for design validation" — i.e., the diagrams a documentation set needs already exist, or nearly exist, as the committed system design's dynamic views, and this step's job is to identify which of those to reuse verbatim versus which new document (an ADR, a runbook, an architecture overview) still needs to be written and sequenced. "Done" means an outline naming every document this activity will produce, which committed diagrams/decisions each one draws on, and where each will live in the repo — nothing is authored yet. This step must NOT invent new architectural facts (components, relationships, decisions) that are not already in the committed system design, must NOT skip drawing an activity diagram for any nested-condition use case the outline covers, and must NOT let the outline stand in for the documents themselves.

## Steps

> **State changes go through the `aiarch-state` MCP tools, not hand-edits.** Where a step below says to record a service contract, phase artifact, or testing artifact and "commit onto branch", do it with the matching tool — `recordServiceContract` / `recordPhaseArtifact` / `recordTestingState` — and finish with `publishDraft`. Do **not** hand-edit `.aiarch/state/project.json` or run `git` for state; only source/doc **files** (code, docs) are git-committed by you. See [[the-method-project-state]].

1. **Read what you need** from `.aiarch/state/project.json` per [[the-method-project-state]]: the activity, the committed system design (components, relationships, and the dynamic view/call chain per core use case), and the naming and layering decisions those diagrams already encode.
2. **Produce** the phase artifact: a documentation outline — the list of documents this activity will author, which committed diagrams or decisions each one reuses or references versus what is new, and each document's intended location under the repo's `docs/` conventions — recorded to `.aiarch/state/project.json`'s phase-artifacts store via `recordPhaseArtifact` per [[the-method-project-state]], then committed onto branch `activity/<activity_id>`.
3. **Verify** (only your own output; fast checks): every diagram or decision the outline cites actually exists in the committed system design; every core use case the outline covers that has a nested condition is slated for an activity diagram, not prose alone; no document in the outline references a component or relationship absent from the committed design.
4. **Stop.** Do not mark phase status (the Manager owns that) and do not merge. Leave the PR open for the gate.
