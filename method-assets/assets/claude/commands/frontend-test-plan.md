# /frontend-test-plan

> The Flows step of a frontend activity: enumerate every way this UI surface's flows could fail to satisfy its approved design and the contracts it calls, before construction starts.

**Arguments** — `$ARGUMENTS` is `<component_id> <activity_id>` (two space-separated tokens; `component_id` may be empty for non-component activities). Parse once; do not swap them. Work lands on the shared activity branch `activity/<activity_id>` and its single PR — you are contributing commits to a PR that already exists (or open it if this is the first phase). Do NOT open a second PR.

**Agent + skills.** Work to the standard of the **`test-engineer`** agent (`.claude/agents/test-engineer.md`). Follow **[[the-method-testing]]** and **[[the-method-project-state]]** for all reading/updating of `.aiarch/state/project.json`.

**Goal / intention.** Per ch. 9, a test engineer is "not a tester, but a full-fledged software engineer" whose code exists to break the system — a higher caliber of engineer than the role's title suggests, and that same caliber of attention, not a checklist, is what this surface's flow-level test plan requires. Per ch. 11, this is not incidental busywork bolted onto UI work: in Löwy's own worked project-design example, the Client's up-front, high-float activities are explicitly requirements, test plan, and UI design — a frontend surface's test plan is scheduled and floated exactly like a service's, deliberately early, so it exists before construction rather than being squeezed in afterward. The same chapter shows what happens when that discipline slips: deferring a test-plan/test-harness activity in Löwy's example consumed 77% of its float, turning a low-risk enabler into one of the riskiest activities in the project. For this step, the test-engineer is not judging the UI design concept's aesthetics — enumerate every way this surface's flows (its journeys through the core use cases it supports) could fail to satisfy the approved UI design and the frozen contracts of the Managers/Engines those flows call. "Done" means that enumeration exists and traces to real flows and real contract operations, not that any test code has been written. This step must NOT write the surface's implementation, must NOT stop at isolated component/widget checks, and must NOT invent failure modes disconnected from an actual flow or contract.

## Steps

1. **Read what you need** from `.aiarch/state/project.json` per [[the-method-project-state]]: the activity, this surface's approved UI design concept, the core use cases and persona journeys it serves, and the frozen contracts of the Manager/Engine components its flows call.
2. **Produce** the phase artifact: this surface's flows test-plan slice — the enumerated ways each flow could fail to satisfy the approved UI design or the contracts it depends on, traced to specific flows and contract operations — recorded into the phase-artifacts store via `recordPhaseArtifact` per [[the-method-project-state]] and committed onto branch `activity/<activity_id>`.
3. **Verify** (only your own output; fast checks): every item in the plan traces to a real flow named in the UI design or a real operation on a called component's frozen contract — no speculative failure modes; the plan contains no implementation or test code, only the plan.
4. **Stop.** Do not mark phase status (the Manager owns that) and do not merge. Leave the PR open for the gate.
