# /testing-perf-construction

> The Rig Construction step of a testing-perf activity: build the performance test rig the Perf Scenario Design specified, and add your commits to the activity PR.

**Arguments** — `$ARGUMENTS` is `<component_id> <activity_id>` (two space-separated tokens; `component_id` is typically empty — the performance test rig is a system-level activity, not a per-component one). Parse once; do not swap them. Work lands on the shared activity branch `activity/<activity_id>` and its single PR — you are contributing commits to a PR that already exists (or open it if this is the first phase). Do NOT open a second PR.

**Agent + skills.** Work to the standard of the **`test-engineer`** agent (`.claude/agents/test-engineer.md`). Follow **[[the-method-testing]]** and **[[the-method-project-state]]** for all reading/updating of `.aiarch/state/project.json`.

**Goal / intention.** As with Perf Scenario Design, this is grounded honestly: Löwy's two Project Design worked examples (ch. 11, ch. 13) contain no performance-testing activity, so there is no worked example of "how a perf rig gets built" to cite. What does apply is ch. 9's definition of the test-engineer role — the same role that "designs test rigs, simulators, automation" for the System Test Harness builds this rig too, and it is judged by the same discipline: it must break the system, not vouch for it. Per this repo's own binding test-authoring constitution (the-method-testing §7, which this rig is subject to exactly as the System Test Harness is), the load-bearing rig is wire-level and black-box against the system's published Client-layer surfaces — a separate module, sibling to the server, outside its package tree, importing zero server code and no mocking library, so Go's `internal/` rule compiler-seals the system's internals against it. The test-engineer that authors it stays blind to any component's implementation, seeing only the core use cases and the published Client-surface contracts, for the same anti-cheat reason the harness author does. This step turns the frozen Perf Scenario Design into real load-generation code: one flow per designed scenario, driving the design's target surfaces at the design's load profile, checked against the design's threshold. "Done" means the rig builds, its structural checks pass, and every scenario in the design has a corresponding implemented flow; this step must NOT change the scenarios, targets, or thresholds the Detailed Design phase froze (any gap goes back to that phase, not silently patched here), and must NOT execute a full load run and interpret the results as a pass/fail verdict — that is System Testing's job, run later by the `software-tester` against the built system, not this construction step's.

## Steps

> **State changes go through the `aiarch-state` MCP tools, not hand-edits.** Where a step below says to record a service contract, phase artifact, or testing artifact and "commit onto branch", do it with the matching tool — `recordServiceContract` / `recordPhaseArtifact` / `recordTestingState` — and finish with `publishDraft`. Do **not** hand-edit `.aiarch/state/project.json` or run `git` for state; only source/doc **files** (code, docs) are git-committed by you. See [[the-method-project-state]].

1. **Read what you need** from `.aiarch/state/project.json` per [[the-method-project-state]]: the activity, the Perf Scenario Design produced in the Perf Scenario Design phase, and the committed system design's Client-layer surfaces it targets — not any component's internal implementation.
2. **Produce** the phase artifact: the rig code — implementing the design's module boundary (a Go module sibling to the server, outside its package tree, importing zero server code and no mocking libraries), with one load-generation flow per designed scenario driving the design's target Client-layer surfaces over the wire at the design's load profile and threshold — committed onto branch `activity/<activity_id>`; record the rig's reference and status into `.testingState.perfHarness` per [[the-method-project-state]].
3. **Verify YOUR code** (working directory the rig module's own root): `gofmt -w .`; `GOWORK=off go build ./...`; `GOWORK=off go vet ./...`; the module's own no-infra structural gate (`make test-short`, or equivalent) passes, confirming no import of the server module or a mocking library and no placement inside the server's package tree; every scenario in the Perf Scenario Design has a corresponding implemented flow.
4. **Stop.** Do not mark phase status (the Manager owns it) and do not merge. Leave the PR for the gate.
