# /testing-systemtest-construction

> The Use-Case Execution step of a testing-systemtest activity: drive every core use case in the System Test Plan through the integrated system via the System Test Harness, and file a defect for every break.

**Arguments** — `$ARGUMENTS` is `<component_id> <activity_id>` (two space-separated tokens; `component_id` is typically empty — System Testing is a system-level activity, not a per-component one). Parse once; do not swap them. Work lands on the shared activity branch `activity/<activity_id>` and its single PR — you are contributing commits to a PR that already exists (or open it if this is the first phase). Do NOT open a second PR.

**Agent + skills.** Work to the standard of the **`software-tester`** agent (`.claude/agents/software-tester.md`). Follow **[[the-method-testing]]** and **[[the-method-project-state]]** for all reading/updating of `.aiarch/state/project.json`.

**Goal / intention.** Both of Löwy's full worked examples place System Testing at the very tail of the project: ch. 11's Table 11-1 has activity 21, "System Testing," depending on the Test Harness and both Clients being complete; ch. 13's TradeMe table lists "System Testing" (10 days) directly ahead of "System Rollout." This is the activity that occupies that terminal slot — the tester actually driving the plan against the real, integrated, daily-built system rather than a developer's branch. Staffing for it is deliberate: per ch. 11, "one additional tester is required during system testing," and per ch. 13 the same pattern recurs almost verbatim ("one additional tester was required for the system testing activity"); both worked examples also note high-quality construction can remove developers from this activity entirely — ch. 11's planning assumption that "developers will produce such high-quality work that they will not be needed during system testing," echoed in ch. 13's "TradeMe's high quality expectations eliminated the need for developers during system testing." That absence is deliberate, not a gap to fill: this activity is the tester's, not the developer's. Per this project's own testing doctrine ([[the-method-testing]]), System Testing carries zero float — every day it slips is a day the whole project slips — so execution must be systematic rather than exploratory, and every genuine break must be captured immediately as a defect, not left as a mental note. "Done" means every use case in the plan's index has at least one recorded run and every observed failure has a filed defect with reproduction steps; this step must NOT write new test scenarios (that was frozen earlier, in the System Test Plan), must NOT modify the System Test Harness (the test-engineer's artifact), and must NOT fix any code — a tester who finds a way to route around a break by touching the system under test is doing the developer's job, not testing it.

## Steps

> **State changes go through the `aiarch-state` MCP tools, not hand-edits.** Where a step below says to record a service contract, phase artifact, or testing artifact and "commit onto branch", do it with the matching tool — `recordServiceContract` / `recordPhaseArtifact` / `recordTestingState` — and finish with `publishDraft`. Do **not** hand-edit `.aiarch/state/project.json` or run `git` for state; only source/doc **files** (code, docs) are git-committed by you. See [[the-method-project-state]].

1. **Read what you need** from `.aiarch/state/project.json` per [[the-method-project-state]]: the activity, the full scenario list and use-case index of the approved System Test Plan, the approved System Test Harness, and the smoke-pass run recorded in the Smoke Pass phase confirming readiness to proceed.
2. **Produce** the phase artifact: drive every use case and scenario in the plan through the harness against the integrated system, exercising it the way a client does (its public surfaces, not internal service calls) — record the run(s) into the testing-state's test-run history and file a defect with reproduction steps for every observed failure into the testing-state's defect list per [[the-method-project-state]]; commit onto branch `activity/<activity_id>`.
3. **Verify** (only your own output; fast checks): every use case in the plan's index has at least one recorded run; every failed run has a corresponding filed defect — a non-deterministic failure is a defect to file, not noise to ignore; no run was recorded against anything outside the approved plan; no product code was changed.
4. **Stop.** Do not mark phase status (the Manager owns that) and do not merge. Leave the PR open for the gate.
