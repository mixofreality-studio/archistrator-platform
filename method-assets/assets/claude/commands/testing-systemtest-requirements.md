# /testing-systemtest-requirements

> The Smoke Pass step of a testing-systemtest activity: before spending any of System Testing's zero-float schedule on full use-case execution, confirm the System Test Harness actually connects to the freshly integrated system and its plumbing is alive.

**Arguments** — `$ARGUMENTS` is `<component_id> <activity_id>` (two space-separated tokens; `component_id` is typically empty — System Testing is a system-level activity, not a per-component one). Parse once; do not swap them. Work lands on the shared activity branch `activity/<activity_id>` and its single PR — you are contributing commits to a PR that already exists (or open it if this is the first phase). Do NOT open a second PR.

**Agent + skills.** Work to the standard of the **`software-tester`** agent (`.claude/agents/software-tester.md`). Follow **[[the-method-testing]]** and **[[the-method-project-state]]** for all reading/updating of `.aiarch/state/project.json`.

**Goal / intention.** Per ch. 9, "modern software systems need configuration management, deployment scripts, a build process, daily builds, and smoke tests" — the practice of exercising the plumbing before trusting deeper results — though ch. 9 lumps the continuous, daily version of that practice "under DevOps." This step is a distinct, one-time application of the same idea, done by the tester rather than DevOps, at the specific moment System Testing (`N-IT`) opens: the System Test Plan and System Test Harness are both approved and the system is freshly integrated, but nobody has yet confirmed the harness actually talks to this build. Per ch. 11's and ch. 13's worked examples, staffing for this activity is deliberately split in two: "one tester is required from the start of construction of the system services until the end of testing" (ch. 11) plus "one additional tester ... during system testing" (echoed in ch. 13's "a single quality control tester was required from the start of construction to the end of the project ... one additional tester was required for the system testing activity") — meaning a second tester is joining right now, and the Smoke Pass is the natural first joint act: prove the harness runs against the real system before committing the ratio-driven tester effort (ch. 9: "changing the ratio of testers to developers, such as 1:1 or even 2:1 ... allows the developers to spend less time testing") to the full use-case pass. Both worked examples also place System Testing at the very tail of the network, gated on the Test Harness and every Client being done (ch. 11's Table 11-1: activity 21 depends on 5, 19, 20) — this step is where the tester confirms that gate is met in practice, not just on paper. "Done" means the harness connects and each core Client-layer surface in the plan's use-case index responds; it does NOT mean running the full test plan (that is Use-Case Execution's job next) and must NOT silently wave through a hard connectivity failure.

## Steps

1. **Read what you need** from `.aiarch/state/project.json` per [[the-method-project-state]]: the activity, the approved System Test Plan (its use-case index and scenario list), and the approved System Test Harness. Confirm both carry an approved status before proceeding — a smoke pass against an unapproved plan or harness is premature.
2. **Produce** the phase artifact: a smoke-pass test run — one quick pass invoking the harness against each core Client-layer surface named in the plan's use-case index to confirm the integrated system responds, recorded via `recordTestingState` as an entry in the testing-state's test-run history per [[the-method-project-state]] and committed onto branch `activity/<activity_id>`. If the harness fails to connect or a surface is unreachable, file that as a defect in the testing-state's defect list rather than leaving it unrecorded.
3. **Verify** (only your own output; fast checks): the run references the actual approved plan and harness (not an invented one); the pass covers connectivity/liveness only, not every scenario in the plan; any hard failure was filed as a defect.
4. **Stop.** Do not mark phase status (the Manager owns that) and do not merge. Leave the PR open for the gate.
