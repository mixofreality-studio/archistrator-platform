# /testing-harness-detailed-design

> The Harness Design step of a testing-harness activity: freeze the System Test Harness's module boundary, target surfaces, and use-case organization before any harness code is written.

**Arguments** — `$ARGUMENTS` is `<component_id> <activity_id>` (two space-separated tokens; `component_id` is typically empty — the System Test Harness is a system-level activity, not a per-component one). Parse once; do not swap them. Work lands on the shared activity branch `activity/<activity_id>` and its single PR — you are contributing commits to a PR that already exists (or open it if this is the first phase). Do NOT open a second PR.

**Agent + skills.** Work to the standard of the **`test-engineer`** agent (`.claude/agents/test-engineer.md`). Follow **[[the-method-testing]]** and **[[the-method-project-state]]** for all reading/updating of `.aiarch/state/project.json`.

**Goal / intention.** Per ch. 14, "the project must have an explicit activity in which qualified test engineers develop a comprehensive test harness" as the System Test Harness. Per ch. 11's worked example, this activity is scheduled early, right alongside the System Test Plan, and deliberately carries very high float — 65 days in Löwy's own case — because it depends on little else being finished; deferring it instead consumes 77% of that float, turning a low-risk enabler into one of the riskiest activities in the project, which is why his Caution insists "every software project should have a test engineer" and that test engineers be staffed before extra developers. Before writing a line of harness code, that early slot is best spent the same way a service's Detailed Design freezes a contract before Construction begins: deciding the harness's module boundary (a Go module sibling to the server, outside its package tree, importing zero server code — per the-method-testing's binding constitution, §7), which of the system's Client-layer surfaces it will drive and how, and how each scenario in the just-authored System Test Plan maps onto a use-case-organized test flow. The test engineer does this blind to any component's internal implementation — per §7 R5, seeing only the committed core use cases and the system's published Client-surface contracts, never component source — because the harness must break the system from the outside, not vouch for its internals. "Done" means every System Test Plan scenario has a planned flow and a target surface; this step must NOT write harness code (that is Harness Construction's job) and must NOT reach into any component's implementation to decide the design.

## Steps

> **State changes go through the `aiarch-state` MCP tools, not hand-edits.** Where a step below says to record a service contract, phase artifact, or testing artifact and "commit onto branch", do it with the matching tool — `recordServiceContract` / `recordPhaseArtifact` / `recordTestingState` — and finish with `publishDraft`. Do **not** hand-edit `.aiarch/state/project.json` or run `git` for state; only source/doc **files** (code, docs) are git-committed by you. See [[the-method-project-state]].

1. **Read what you need** from `.aiarch/state/project.json` per [[the-method-project-state]]: the activity, the committed System Test Plan's scenarios, the committed core use cases, and the committed system design's Client-layer surfaces and their published contracts — not any component's internal implementation.
2. **Produce** the phase artifact: the Harness Design — the module boundary and placement (a Go module sibling to the server, outside its package tree, importing zero server code or mocking libraries), the target Client-layer surfaces the harness will drive and how, and the mapping from each System Test Plan scenario to a planned, use-case-organized test flow — recorded into the phase-artifacts store per [[the-method-project-state]] and committed onto branch `activity/<activity_id>`.
3. **Verify** (only your own output; fast checks): every scenario in the System Test Plan maps to a planned flow and a target surface; the design places the harness outside the server's package tree with zero server-code imports specified; the design references only committed Client-surface contracts, never a component's internal implementation; no harness code has been written.
4. **Stop.** Do not mark phase status (the Manager owns that) and do not merge. Leave the PR open for the gate.
