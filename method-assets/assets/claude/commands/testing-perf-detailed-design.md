# /testing-perf-detailed-design

> The Perf Scenario Design step of a testing-perf activity: freeze the performance test rig's target scenarios, load profiles, and pass/fail thresholds before any rig code is written.

**Arguments** — `$ARGUMENTS` is `<component_id> <activity_id>` (two space-separated tokens; `component_id` is typically empty — the performance test rig is a system-level activity, not a per-component one). Parse once; do not swap them. Work lands on the shared activity branch `activity/<activity_id>` and its single PR — you are contributing commits to a PR that already exists (or open it if this is the first phase). Do NOT open a second PR.

**Agent + skills.** Work to the standard of the **`test-engineer`** agent (`.claude/agents/test-engineer.md`). Follow **[[the-method-testing]]** and **[[the-method-project-state]]** for all reading/updating of `.aiarch/state/project.json`.

**Goal / intention.** Löwy's Method has no dedicated performance-testing chapter, and this is stated honestly rather than papered over: ch. 11 and ch. 13 are the two full Project Design worked examples in the book, and neither example's activity list or network contains a performance-test activity — the search across both chapters turns up nothing under "performance," "load test," or "throughput" that refers to system load testing (the one "throughput" discussion in ch. 11 is about a compressed *team's* productivity, not the system under test). What genuinely grounds this step: ch. 9's definition of the test-engineer role states plainly that test engineers are engaged in "developing fake communication channels; designing and developing regression testing; and designing test rigs, simulators, automation, and more" — a performance test rig is one instance of that named class of artifact, owned by the same role that owns the System Test Harness. And both worked examples do establish a structural placement pattern for test-engineer artifacts: the Test Plan and Test Harness are scheduled early with very high float because they depend on little else being finished, while System Testing sits at or near the terminal end of the critical path because it needs a real, integrated system to exercise. A performance rig is structurally like System Testing, not like the Test Harness — it needs a system worth loading — which is exactly why this project's own testing-state schema classifies the performance rig (N-PERF) as a terminal, test-engineer-owned artifact rather than an early high-float one. This step is the Detailed-Design analogue for that rig: before any load-generation code exists, decide what to load (the workload mix, organized by core use case), how hard (a load profile — ramp/steady/spike as applicable), and against which of the system's Client-layer surfaces, plus a numeric pass/fail threshold per scenario. "Done" means every scenario has a target surface, a load profile, and a threshold; this step must NOT write rig code, and must NOT invent load targets from nothing — ground them in the project's own declared usage assumptions where available, and mark anything without that grounding as an explicit assumption rather than a fabricated number.

## Steps

> **State changes go through the `aiarch-state` MCP tools, not hand-edits.** Where a step below says to record a service contract, phase artifact, or testing artifact and "commit onto branch", do it with the matching tool — `recordServiceContract` / `recordPhaseArtifact` / `recordTestingState` — and finish with `publishDraft`. Do **not** hand-edit `.aiarch/state/project.json` or run `git` for state; only source/doc **files** (code, docs) are git-committed by you. See [[the-method-project-state]].

1. **Read what you need** from `.aiarch/state/project.json` per [[the-method-project-state]]: the activity, the committed system design's Client-layer surfaces and core use cases, the Planning Assumptions' declared usage (expected daily active users, requests per minute, average payload size) as the basis for load targets, and any already-approved System Test Plan scenarios so the perf scenarios align with real use cases instead of an invented traffic pattern.
2. **Produce** the phase artifact: the Perf Scenario Design — a scenario list where each entry names a target Client-layer surface, a workload/use-case mix, a load profile, and a numeric pass/fail threshold (e.g., a latency percentile, an error-rate ceiling, or a throughput floor) — grounded in the declared usage assumptions where they exist and explicitly flagged as an assumption where they don't — recorded into the phase-artifacts store via `recordPhaseArtifact` per [[the-method-project-state]] and committed onto branch `activity/<activity_id>`.
3. **Verify** (only your own output; fast checks): every scenario names a target surface, a load profile, and a numeric threshold; each threshold either traces to a declared-usage figure or is explicitly marked as an assumption; no rig code has been written.
4. **Stop.** Do not mark phase status (the Manager owns that) and do not merge. Leave the PR open for the gate.
