# /testing-qa-construction

> The Process Audit step of a testing-qa activity: audit whether the development process — including the gates defined in Gate Definition — is actually being followed, and tune the process where it is not.

**Arguments** — `$ARGUMENTS` is `<component_id> <activity_id>` (two space-separated tokens; `component_id` is typically empty — this is a project-level activity, not a per-component one). Parse once; do not swap them. Work lands on the shared activity branch `activity/<activity_id>` and its single PR — you are contributing commits to a PR that already exists (or open it if this is the first phase). Do NOT open a second PR.

**Agent + skills.** Work to the standard of the **`qa-engineer`** agent (`.claude/agents/qa-engineer.md`). Follow **[[the-method-testing]]** and **[[the-method-project-state]]** for all reading/updating of `.aiarch/state/project.json`.

**Goal / intention.** Per ch. 9, true QA "typically involves a single, senior expert who answers the question: What will it take to assure quality?" — an answer requiring the QA person to "orient the entire development process to assure quality," to "prevent problems from ever happening," and to "track the root causes of problems and to fix them"; ch. 9 stresses this "has little to do with testing," and that "most teams incorrectly refer to their quality control and testing activities as" QA. Ch. 14 makes the audit concrete: the engaged QA person must "review the development process, tune it to assure quality, and create a process that is both effective and easy to follow," in a way that "support[s] investigation and elimination of the root cause of defects or, even better, ... proactively prevent[s] problems from happening in the first place." Ch. 14 also ties structured, role-based reviews (moderator, owner, scribe, reviewers) at both the service and system level to review effectiveness — the audit should check that this discipline is actually happening, not assume it. This step is that ongoing audit applied against the gates this activity itself defined in its Gate Definition phase: is the process actually being followed (are the defined gates being hit, are reviews structured and happening, is the constant-defect-free-codebase discipline holding), and where it is not, how should the process be tuned. "Done" means an audit note that states what is and is not being followed and what process changes (if any) are recommended. This step must NOT write or review any component's contract, design, or code; must NOT run tests, execute the harness, or file defects (that is software-tester's job, not QA's — see [[the-method-testing]]); and — because "the presence of a QA person is a sign of organizational maturity," this is process oversight, not enforcement — must NOT itself alter the recorded `.reviewPolicy`, which remains a Manager-owned verb the QA agent never writes.

## Steps

> **State changes go through the `aiarch-state` MCP tools, not hand-edits.** Where a step below says to record a service contract, phase artifact, or testing artifact and "commit onto branch", do it with the matching tool — `recordServiceContract` / `recordPhaseArtifact` / `recordTestingState` — and finish with `publishDraft`. Do **not** hand-edit `.aiarch/state/project.json` or run `git` for state; only source/doc **files** (code, docs) are git-committed by you. See [[the-method-project-state]].

1. **Read what you need** from `.aiarch/state/project.json` per [[the-method-project-state]]: the activity, the gate-definition note this activity's Gate Definition phase produced, and evidence of how the process has actually run since (recent phase artifacts, review outcomes, construction status of other activities).
2. **Produce** the phase artifact: the process-audit note — stating which defined gates and review disciplines are being followed, which are not, and what process tuning (if any) is recommended — recorded into the phase-artifacts store per [[the-method-project-state]] and committed onto branch `activity/<activity_id>`. Do NOT write `.reviewPolicy` yourself; that stays a Manager-owned verb, applied separately if the Manager acts on this audit.
3. **Verify** (only your own output; fast checks): every finding cites the specific gate or discipline it checks against and whether it was followed; the note contains no component contract, design, code, test, or defect content; the note does not itself modify the enforced review policy.
4. **Stop.** Do not mark phase status (the Manager owns that) and do not merge. Leave the PR open for the gate.
