# /testing-harness-construction

> The Harness Construction step of a testing-harness activity: build the System Test Harness the Harness Design specified, and add your commits to the activity PR.

**Arguments** — `$ARGUMENTS` is `<component_id> <activity_id>` (two space-separated tokens; `component_id` is typically empty — the System Test Harness is a system-level activity, not a per-component one). Parse once; do not swap them. Work lands on the shared activity branch `activity/<activity_id>` and its single PR — you are contributing commits to a PR that already exists (or open it if this is the first phase). Do NOT open a second PR.

**Agent + skills.** Work to the standard of the **`test-engineer`** agent (`.claude/agents/test-engineer.md`). Follow **[[the-method-testing]]** and **[[the-method-project-state]]** for all reading/updating of `.aiarch/state/project.json`.

**Goal / intention.** Per ch. 14, the project must have an explicit activity in which qualified test engineers "develop a comprehensive test harness" — the code artifact that follows the System Test Plan and the Harness Design. Per ch. 11's worked example, the Test Harness activity is scheduled early alongside the Test Plan and deliberately carries very high float; building it now, while that float is intact, is what preserves the benefit, since deferring it instead consumes 77% of the float and turns a low-risk enabler into one of the riskiest activities in the project — Löwy's Caution that "every software project should have a test engineer" is aimed squarely at not letting this slip. Per ch. 9, having this "anti-system" ready early "does wonders for quality because you can discover problems as soon as they occur, isolate root causes, avoid ripple effects of changes, [and] eliminate superimposition of defects masking other defects." This step turns the frozen Harness Design into real code: the module the design specified, built to drive the system's Client-layer surfaces over the wire, organized by the core use cases the System Test Plan traced, touching zero of the system's internal implementation — the test engineer builds to break the system, not to vouch for its internals. "Done" means the harness builds and its structural checks pass against the design's target surfaces and use-case organization; this step must NOT weaken the design's module boundary (no server-code imports, no mocking library standing in for the real system) and must NOT expand into System Testing itself — running the harness to find and file real defects is the terminal System Testing activity's job (`software-tester`), not this one.

## Steps

> **State changes go through the `aiarch-state` MCP tools, not hand-edits.** Where a step below says to record a service contract, phase artifact, or testing artifact and "commit onto branch", do it with the matching tool — `recordServiceContract` / `recordPhaseArtifact` / `recordTestingState` — and finish with `publishDraft`. Do **not** hand-edit `.aiarch/state/project.json` or run `git` for state; only source/doc **files** (code, docs) are git-committed by you. See [[the-method-project-state]].

1. **Read what you need** from `.aiarch/state/project.json` per [[the-method-project-state]]: the activity, the Harness Design produced in the Harness Design phase, and the System Test Plan scenarios it targets.
2. **Produce** the phase artifact: the harness code — implementing the design's module boundary (a Go module sibling to the server, outside its package tree, importing zero server code and no mocking libraries), organized by core use case, driving the design's target Client-layer surfaces over the wire — committed onto branch `activity/<activity_id>`; record the module's reference and status into `.testingState.harnessModule` per [[the-method-project-state]].
3. **Verify YOUR code** (working directory the harness module's own root, e.g. `systemtests`): `gofmt -w .`; `GOWORK=off go build ./...`; `GOWORK=off go vet ./...`; the module's own no-infra structural gate (`make test-short`, or equivalent) passes, confirming no import of the server module or a mocking library and no placement inside the server's package tree.
4. **Stop.** Do not mark phase status (the Manager owns it) and do not merge. Leave the PR for the gate.
