# /testing-plan-construction

> The Plan Authoring step of a testing-plan activity: enumerate every way the integrated system could fail, across the use cases and call chains already traced, as the System Test Plan.

**Arguments** — `$ARGUMENTS` is `<component_id> <activity_id>` (two space-separated tokens; `component_id` is typically empty — this is a system-level activity, not a per-component one). Parse once; do not swap them. Work lands on the shared activity branch `activity/<activity_id>` and its single PR — you are contributing commits to a PR that already exists (or open it if this is the first phase). Do NOT open a second PR.

**Agent + skills.** Work to the standard of the **`test-engineer`** agent (`.claude/agents/test-engineer.md`). Follow **[[the-method-testing]]** and **[[the-method-project-state]]** for all reading/updating of `.aiarch/state/project.json`.

**Goal / intention.** Per ch. 9, test engineers are "a higher caliber of engineer than regular software engineers, because writing test engineering code often involves more difficult tasks: developing fake communication channels; designing and developing regression testing; and designing test rigs, simulators, automation, and more" — and having such an "anti-system" system ready "does wonders for quality because you can discover problems as soon as they occur, isolate root causes, avoid ripple effects of changes, [and] eliminate superimposition of defects masking other defects." Per ch. 14, the deliverable of this activity is explicit: "a list of all the ways to break the system and prove it does not work." This is the authoring step itself — using the use-case trace already captured in the prior phase, the test engineer now enumerates, for every core use case and call chain in scope, the concrete ways the integrated system could fail to satisfy it. Per ch. 11, this activity is scheduled early with deliberately high float so it can be done thoroughly now rather than squeezed in late once its float has been consumed by delay. Per the-method-testing's test-plan-first discipline (not TDD, no BDD/Gherkin), this plan precedes and will drive the later System Test Harness that actually executes it; unit testing alone is "borderline useless" (ch. 2), so the plan must target the integrated system's interactions, not per-component internals. "Done" means every enumerated way-to-break traces to a real use case or call chain from the prior trace; this step must NOT contain harness or test code (that is the System Test Harness activity's job) and must NOT be scoped down to unit-level checks alone.

## Steps

> **State changes go through the `aiarch-state` MCP tools, not hand-edits.** Where a step below says to record a service contract, phase artifact, or testing artifact and "commit onto branch", do it with the matching tool — `recordServiceContract` / `recordPhaseArtifact` / `recordTestingState` — and finish with `publishDraft`. Do **not** hand-edit `.aiarch/state/project.json` or run `git` for state; only source/doc **files** (code, docs) are git-committed by you. See [[the-method-project-state]].

1. **Read what you need** from `.aiarch/state/project.json` per [[the-method-project-state]]: the activity, the use-case trace produced in the Requirements phase, the committed system design, and the committed core use cases.
2. **Produce** the phase artifact: the System Test Plan — the enumerated ways the integrated system could fail to satisfy each traced use case and call chain — recorded via `recordTestingState` to `.testingState.systemTestPlan` per [[the-method-project-state]] and committed onto branch `activity/<activity_id>`.
3. **Verify** (only your own output; fast checks): every item in the plan traces to a real core use case or call chain from the use-case trace; the plan contains no harness or test code and is not scoped to unit-level checks only; nothing outside the traced scope was invented.
4. **Stop.** Do not mark phase status (the Manager owns that) and do not merge. Leave the PR open for the gate — the plan still awaits its `test-plan` review (system-architect + product-manager + qa-engineer, per [[the-method-review-routing]]) before it is relied upon.
