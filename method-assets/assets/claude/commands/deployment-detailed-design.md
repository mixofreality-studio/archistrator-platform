# /deployment-detailed-design

> The Provisioning Spec step of a deployment activity: capture the explicit infrastructure and environment planning assumptions this project's deployment depends on, before any provisioning work is built against them.

**Arguments** — `$ARGUMENTS` is `<component_id> <activity_id>` (two space-separated tokens; `component_id` may be empty for non-component activities). Parse once; do not swap them. Work lands on the shared activity branch `activity/<activity_id>` and its single PR — you are contributing commits to a PR that already exists (or open it if this is the first phase). Do NOT open a second PR.

**Agent + skills.** Work to the standard of the **`project-manager`** agent (`.claude/agents/project-manager.md`). Follow **[[the-method-planning-assumptions]]** and **[[the-method-project-state]]** for all reading/updating of `.aiarch/state/project.json`.

**Goal / intention.** Per ch. 7, critical path analysis — and therefore the whole project design — cannot proceed without planning assumptions: "you must know the resources available for the project... The planning assumptions will include which type of resource is required at which phase of the project." Appendix C's Project Design Guidelines list "capture and verify planning assumptions" as a general, non-optional rule, on par with never designing a project without an architecture. Löwy's own cost model treats Build and DevOps as a distinct, budgeted resource line alongside the core team and testers — infrastructure and provisioning are real project resources, not free byproducts of coding. This step is the deployment activity's equivalent of a service's detailed design: before anything is provisioned, write down, explicitly and in project-design terms, what environments/regions/infra kind this deployment assumes, what resources (compute, network, secrets, third-party services) it depends on, and at which phase of the project each is needed — exactly the planning-assumption content ch. 7 says the rest of the network's timing depends on. Löwy does not describe a "cloud provisioning" activity by name; the book's own devops references are the cost-model resource line in ch. 7 and the general planning-assumptions guideline in Appendix C, so this step is grounded in that principle, not in a book chapter about infrastructure design specifically. "Done" means a written provisioning spec a junior developer can build infra/manifests against without guessing. This step must NOT provision anything itself, must NOT silently assume infrastructure choices the committed system design and operational concepts do not already imply, and must NOT invent staffing or resource numbers — pull them from the committed planning assumptions or flag the gap.

## Steps

> **State changes go through the `aiarch-state` MCP tools, not hand-edits.** Where a step below says to record a service contract, phase artifact, or testing artifact and "commit onto branch", do it with the matching tool — `recordServiceContract` / `recordPhaseArtifact` / `recordTestingState` — and finish with `publishDraft`. Do **not** hand-edit `.aiarch/state/project.json` or run `git` for state; only source/doc **files** (code, docs) are git-committed by you. See [[the-method-project-state]].

1. **Read what you need** from `.aiarch/state/project.json` per [[the-method-project-state]]: the activity, the committed system design's deployment topology (from operational concepts), and the committed planning assumptions (`infrastructureKind`, `declaredUsage`, resources, terms) that this deployment must honor.
2. **Produce** the phase artifact: a provisioning spec — target environment(s), infrastructure kind, required resources and their dependencies, and any environment-specific terms/notes a junior developer needs to build the infra/manifests against — recorded to `.aiarch/state/project.json`'s phase-artifacts store via `recordPhaseArtifact` per [[the-method-project-state]], then committed onto branch `activity/<activity_id>`.
3. **Verify** (only your own output; fast checks): every resource named in the spec traces back to either the committed system design/operational concepts or the committed planning assumptions — no invented infra choices; the spec is concrete enough that construction requires no further clarification.
4. **Stop.** Do not mark phase status (the Manager owns that) and do not merge. Leave the PR open for the gate.
